# QA — Troubleshooting RHACM Applications (GitOps)

Domande e risposte avanzate relative a chapter5/5.7-troubleshooting.md

1) Domanda
Metodo sistematico per individuare se il problema di deploy è a livello Git, ApplicationSet, Placement, Argo CD o runtime?  
Risposta  
Segui la pipeline: controlla commit/branch (Git), validità YAML (CI), ApplicationSet generation (Topology view), Placement selezione, stato Application su Argo CD hub/managed e infine logs del runtime. Usare Topology view come punto di partenza per capire "DOVE".

2) Domanda
Come individuare problemi specifici del Pull model quando si vede “application non presente” nell’Argo CD hub?  
Risposta  
Se Pull model, l’Argo CD di hub non gestisce le Application; verificare Argo CD sui managed cluster, controllare connessione di klusterlet, e assicurarsi che ApplicationSet sia configurato per generare Applications nelle destinazioni corrette.

3) Domanda
Quali sono i segnali che indicano che il problema è dovuto a secret/credential mancanti piuttosto che a template malformati?  
Risposta  
Errori di image pull, eventi di unauthorized, e logs con 401/403 indicano problemi di credential; sintassi YAML/templating solitamente produce parse errors o invalid CRD messages. Controllare eventi del pod e describe dell’Application.

4) Domanda
In caso di deploy fallito su molti cluster contemporaneamente, quale strategia usare per rollback?  
Risposta  
Bloccare reconciliation centralmente (sync windows o disable del generator), revert del commit Git che ha introdotto il cambiamento, applicare rollback in wave, monitorare health e ripristinare gradualmente.

5) Domanda
Come raccogliere artefatti di debug utili per post-mortem su un incidente multicluster (what to collect)?  
Risposta  
Collect: diffs tra manifest attesi/effettivi, logs di ApplicationSet, Argo CD controller logs, events dei cluster target, output di oc/kubectl describe per risorse fallite, metriche di reconciliation, e timeline Git commit/CI.
